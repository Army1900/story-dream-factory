# 故事梦工厂（Story Dream Factory）— 系统设计

**日期**：2026-07-11
**状态**：草案 v2，待审查
**作者**：Army1900 + Claude（brainstorming）

---

## 1. 项目愿景

构建一个 Web 系统，让用户定义故事的世界背景与规则、安排角色，然后由**角色 Agent 自主行动**，在「物理决策引擎」（硬规则）与「世界意识 Agent」（软叙事）的共同决策下，让故事在世界中**涌现、演化**。用户作为「导演」可随时介入，观察整个故事的走向。

### 三个核心体验（缺一不可，彼此构成张力）

1. **观察涌现** — 角色自主行动，产生不可预测的关系、合作、冲突。
2. **生成可读故事** — 涌现的事件被织成连贯、有文学性的叙述文本。
3. **互动导演戏剧** — 用户作为导演可随时介入：注入事件、改目标、改规则。

这三者的张力（"完全自主涌现" vs "导演可控" vs "文学可读"）是整个设计的核心约束。

---

## 2. 关键决策汇总

| 维度 | 决策 |
|------|------|
| 模拟骨架 | 抽象地点 + 关系网络 + 事件流；物理引擎管硬规则，世界意识管软叙事 |
| 推进方式 | 自动播放 + 随时暂停介入 |
| 架构范式 | 方案 C 混合分层：角色层（涌现）+ 物理引擎（硬规则）+ 世界意识（软叙事） |
| **世界初始化** | **模板起点 + 7 阶段对话（世界构建助手）+ 定稿健康度检查** |
| LLM 策略 | 多来源、分层路由（世界意识强 / 主角中 / 背景便宜本地）；MVP 先统一跑通，架构预留网关 |
| 技术栈 | Python（FastAPI）+ React（TypeScript） |
| LLM 网关 | LiteLLM（统一多 provider + fallback + 限流 + 成本统计） |
| 持久化（MVP） | SQLite + 内存向量检索（架构隔离，未来迁 PostgreSQL + pgvector） |

```
│  │  Agent 层    │  │  物理引擎    │  │  LLM 网关(分层)   │   │
│  │ 角色 Agent  │  │  (硬规则裁决)│  │ 强/中/弱 + 本地   │   │

│  ┌────────────────────────────────────────────────────┐   │

```

**核心设计原则**：所有组件之间**只通过结构化数据通信**（Event、状态对象），不传递原始 LLM 散文输出。这是让"多模型分层路由"可行的关键——不同模型的风格差异被接口隔离，不会互相传染。

## 4. 核心数据模型

整个系统的"名词"，所有组件围绕它们运作。字段以 Pydantic 风格示意（后端即 FastAPI + Pydantic）。

### World（世界） — 一个模拟实例的根
- `id`, `name`
- `vision`：愿景/前提（短文本，如「一个魔法衰落中的王国，旧秩序崩塌」）
- `setting`：世界观长文本（时代、基调、魔法/科技水平……）
- `rules`：结构化的世界规则列表（如「魔法稀有」「暴力行为会被通缉」）—— 物理引擎和世界意识都读取它
- `clock`：模拟时钟（`tick` 整数 + 虚拟日期），驱动自动播放
- `state_flags`：全局状态标志（如 `war_started=True`），可被事件或导演改变
- `initial_state`：开场态势 —— 含 `opening_situation`（开场瞬间世界状态描述）+ `inciting_event`（引爆事件，模拟开拍时作为 tick 0 注入的第一个事件）
- `model_config`：本世界的 LLM 分层模型映射（不含密钥）

### WorldTemplate（世界模板） — 预设起点
- `id`, `name`, `genre`（如「中世纪奇幻」「赛博朋克」「宫廷权谋」）
- `description`
- `vision_draft`, `setting_draft`, `rules_draft`, `locations_draft`, `characters_draft`：预设草案，作为 7 阶段对话的起点（用户在此基础上改造，而非从零开始）
- MVP 先内置 2-3 个模板

### Location（地点） — 抽象地图节点（无坐标）
- `id`, `name`, `description`
- `neighbors`：连通的相邻地点 id（角色在地点之间移动）
- `occupants`：当前在场角色 id 列表
- `resources`：可交互对象/资源

### Character（角色）
- `id`, `name`, `archetype`（原型）
- `personality`：性格（大五人格维度或标签集）
- `backstory`, `skills`
- `goals`：短期 + 长期目标（**导演可修改**）
- `state`：`location_id`、`health`、`resources`、`mood`、状态标志
- `relationships`：对其他角色的关系（见 Relationship）
- `memory`：记忆流（Agent 私有，见第 7 节）—— 世界构建阶段会写入初始观察（基于 backstory）

### Relationship（关系边） — 角色之间的有向边
- `from_character_id`, `to_character_id`
- `affinity`：好感度数值
- `trust`：信任度数值
- `history`：关键事件引用列表
- 关系网络是「涌现」的核心观测对象，前端可可视化为图谱

### Event（事件） — 事件流的基本单位，**整个系统的流通货币**
- `id`, `tick`, `timestamp`
- `type`：`action` / `dialogue` / `environment` / `conflict` / `relationship_change` / `director`（导演注入）/ `inciting`（引爆事件）
- `participants`：参与角色 id 列表
- `location_id`
- `payload`：结构化负载（行动描述、结果数据）
- `narration`：世界意识生成的文学化叙述文本（可异步后补）
- `visibility`：知晓此事的角色 id 列表（决定谁能写入记忆）

### Memory（记忆条目） — 角色 Agent 的私有记忆流
- `type`：`observation` / `reflection` / `plan`
- `content`：文本
- `timestamp`, `tick`
- `importance`：重要性分（1-10）
- `embedding`：向量（用于检索）

### DirectorDirective（导演指令） — 用户运行时介入（见第 9 节）
- `type`：`inject_event` / `set_goal` / `modify_world` / `force_action`
- `payload`, `target`, `effective_tick`

## 5. 世界初始化流程（世界构建助手）

世界的质量直接决定模拟质量——背景/规则若模糊或自相矛盾，涌现会一团糟。因此世界初始化不是一张表单，而是**一个和 AI 协作、有严格阶段、讨论充分后才定稿**的流程。

### 5.1 关键架构区分：世界构建助手 ≠ 世界意识

| | 世界构建助手（World-Builder） | 世界意识（World-Consciousness） |
|---|---|---|
| **何时运行** | 模拟**开始前**，一次性（可迭代回退） | 模拟**运行中**，持续 |
| **对话对象** | 和**导演**协作 | 不与导演对话，仲裁角色间的世界 |
| **职责** | 把模糊想法变成**清晰、无矛盾、可运行**的世界定义 | 运行时裁判/叙述/一致性 |
| **产物** | 结构化 `World` 对象（含 setting/rules/locations/characters/initial_state） | 实时事件与叙述 |

它俩是**不同的 Agent**，但共享同一份世界知识。世界构建助手的产物，就是世界意识的"宪法"。世界构建助手使用 Tier 1 强模型（世界质量是天花板）。

### 5.2 七阶段漏斗（从抽象到具体，逐层收敛）

每一层为下一层提供约束，越往下越具体：

```
1. 愿景 Vision     "你想讲一个什么样的故事？" → 类型/基调/主题/规模
2. 世界观 Setting   时代/世界设定/核心矛盾（这个世界最大的张力是什么）
3. 规则 Rules      ⭐ 关键：把模糊想法变成【清晰、可执行、无矛盾】的世界规则
                    ——这些规则直接喂给物理引擎和世界意识作为约束
4. 地点 Locations   关键地点 + 抽象连通地图
5. 角色 Characters  性格/背景/目标/初始关系——刻意埋下【张力】（目标冲突、关系纠葛）
                    因为张力是涌现的燃料
6. 开场 Inciting    初始态势 + 引爆事件（打破平衡启动故事，否则开局平淡）
7. 定稿 Finalize    汇总成结构化 World + 一致性健康检查 + 导演确认 → 开拍
```

### 5.3 起点：模板 + 对话

导演先从 2-3 个预设模板（如「中世纪奇幻」「宫廷权谋」「赛博朋克」）中选一个，7 阶段对话以模板草案为起点进行改造，而非从零开始。降低冷启动门槛。

### 5.4 每阶段的交互模式（混合式："自由对话 + 后台清单 + 阶段小结"）

- **自由对话**：导演在阶段内和 AI 自由聊（自然、有创造性）。
- **后台清单**：AI 用一份"该阶段必覆盖维度"清单在后台追踪，主动把话题引向未覆盖维度，确保不遗漏。
- **阶段小结确认**：每阶段结束前，AI 把对话**结构化复述**（"我理解的是……对吗？"），导演确认后才进下一阶段。
- **可回退**：任何阶段都能退回去改（包括定稿后回来改某一层）。
- **矛盾检测**：AI 在阶段 3（规则）和阶段 7（定稿）主动检测矛盾（规则 A 与规则 B 冲突？角色目标与世界观不符？），提示导演解决。

### 5.5 定稿时的"世界健康度检查"

AI 在阶段 7 自动跑一次体检，给出报告：
- 规则是否无矛盾、可执行
- 角色目标是否可触发、是否有足够张力
- 地点连通是否合理
- 有无"死局"风险（某角色完全无法参与故事）

体检通过才建议开拍；有问题则指出并建议回退到对应阶段修复。

### 5.6 产物

定稿后产出**可直接运行的结构化 World 对象**：`vision` + `setting` + `rules` + `locations` + `characters`（含初始 `relationships` 与基于 backstory 的初始 `memory` 观察）+ `initial_state`（开场态势 + 引爆事件）。引擎开拍时先把 `inciting_event` 作为 tick 0 注入，再进入正常模拟循环。

## 6. 模拟循环（系统的心脏）

模拟引擎跑一个后台异步循环，每个 tick（一个虚拟时间单位，如「半天」）执行五个阶段：

```
 每个 tick:
 ┌─────────────────────────────────────────────────────────────┐
 │ ① 角色层 (并行, 调 LLM)                                       │
 │    每个角色看"自己的世界快照"→ 产出结构化 ActionProposal       │
 │    （意图/动作类型/目标对象/预期/对白）—— 此刻角色互不通信      │
 ├─────────────────────────────────────────────────────────────┤
 │ ② 物理引擎层 (确定性, 不调 LLM)                                │
 │    可行性校验 + 移动 + 资源 + 战斗数值结算 + 关系数值变化       │
 │    + 冲突检测 → 产出 ResolvedAction + 原始 Event(无叙述)       │
 ├─────────────────────────────────────────────────────────────┤
 │ ③ 导演注入点                                                  │
 │    若导演暂停并提交了指令 → 转成 Event 或状态变更, 合并进本步  │
 │    无介入则直通                                               │
 ├─────────────────────────────────────────────────────────────┤
 │ ④ 世界意识层 (调 LLM, 最强模型)                                │
 │    仲裁模糊冲突 + 一致性修正(防离谱) + 叙述生成(文学化文本)     │
 │    + 可选剧情张力注入 → 最终 Event(含 narration)              │
 ├─────────────────────────────────────────────────────────────┤
 │ ⑤ 写回                                                        │
 │    更新世界状态 → 追加事件流 → 按可见性写入各角色记忆          │
 │    → 时钟推进(tick+1) → WebSocket 推送新事件给前端            │
 └─────────────────────────────────────────────────────────────┘
```

### 关键设计点

- **角色层并行、互不通信**：每个角色只产出"提案"，不在此步直接交互——可并行调 LLM，避免顺序依赖。真正的交互结果由物理引擎 + 世界意识统一裁决。
- **物理引擎是确定性、可单测、不烧 token 的**：它是"硬规则可信"的来源。战斗/资源/移动有数值依据，不全靠 LLM 编造。LLM 只在阶段④处理"数值算不了的模糊事"（说服是否成功、感情的微妙转变）。
- **Event 是流通货币**：每一层只消费和产出结构化 Event。

### 自动播放 + 暂停的配合

- 引擎是后台 asyncio 循环，每步之间有可调节奏（0.5×/1×/2×…）。
- 前端随时发「暂停」→ 引擎在**当前 tick 跑完后**停下（不停在半步，保证状态一致）。
- 暂停时导演提交的指令进入队列，在**下一个 tick 的阶段③**注入。
- 「继续」恢复循环。

### 错误处理与降级（LLM 不稳定，必须设计）

| 故障 | 处理 |
|------|------|
| 角色层 LLM 超时/失败 | 网关 fallback 备用模型；仍失败 → 该角色本步"发呆"（默认行动），不阻塞循环 |
| ActionProposal 格式非法 | 解析层重试 + 校验；仍失败 → 降级为"等待" |
| 物理引擎拒绝行动 | 行动失败反馈写入角色记忆（角色会"学到"） |
| 世界意识失败 | 用物理引擎的结构化结果做**降级模板叙述**，不阻塞；标记 narration 待补 |
| 网关速率限制 | 令牌桶限流，超出排队；背景任务路由到本地/便宜模型错峰 |

### 并发与成本

阶段①所有角色并行调 LLM，受网关并发上限约束。分层模型的价值在此：主角走商业 API、背景走本地，错峰避免限流。

## 7. Agent 内部结构

设计参考 Stanford Generative Agents 的记忆流范式，按三层架构做了裁剪。

### 角色 Agent 的决策流程（阶段①内部）

```
 每个角色在阶段①内部:
 ┌──────────────────────────────────────────────────────┐
 │ 1. 感知 Perceive                                      │
 │    从世界快照提取"我能感知的"：所在地点、在场角色、     │
 │    近期公开事件、自身状态（健康/资源/情绪）            │
 ├──────────────────────────────────────────────────────┤
 │ 2. 记忆检索 Retrieve                                  │
 │    score = α·recency + β·importance + γ·relevance     │
 │    （最近性 + 重要性 + 与当下情境的语义相关性）→ top-K   │
 ├──────────────────────────────────────────────────────┤
 │ 3. 反思 Reflect  (周期性, 非每步——MVP 第二期)          │
 │    记忆积累到阈值时, 生成高层洞察:"我发现X一直在骗我"    │
 │    反思本身也是记忆条目, 让角色"长记性"                 │
 ├──────────────────────────────────────────────────────┤
 │ 4. 规划 Plan                                          │
 │    性格 + 目标 + 检索到的记忆 + 当前感知 → 决定下一步    │
 │    维护一个短期计划(可被新事件打断)                     │
 ├──────────────────────────────────────────────────────┤
 │ 5. 产出 ActionProposal (结构化)                        │
 │    意图 / 动作类型 / 目标对象 / 预期 / 对白            │
 └──────────────────────────────────────────────────────┘
```

角色只记住**它自己能感知到的事件**（由 Event 的 `visibility` 字段决定）。

### 核心分工：有限视角 vs 全知视角

| | 角色 Agent | 世界意识 Agent |
|---|---|---|
| **视角** | 第一人称有限视角（有偏见、会犯错、不知全局） | 上帝视角全知（看所有事件、状态、规则） |
| **记忆** | 私有记忆流（只存自己感知到的） | 世界记忆（全局因果链、阵营状态、未解悬念） |
| **作用** | 产生**不可预测的、有性格的行动**（涌现来源） | 保证**一致性与叙述质量**（可读来源） |

**这是整个设计的灵魂**：角色因为"看不全"而做出有偏见的决定，世界意识因为"看得全"而能织出连贯的故事。戏剧性恰恰来自角色的有限理性。

### 世界意识的四个子职责（阶段④内）

| 子职责 | 做什么 | MVP |
|--------|--------|-----|
| **裁判 Arbiter** | 处理物理引擎标记的模糊冲突（两人同时说服一个 NPC、社交后果的微妙判定）→ 结构化裁决 | ✅ 做 |
| **一致性守护 Consistency** | 检查本步有无违反世界规则/角色性格的离谱结果（怯懦者无端英勇冲锋）→ 修正或标记 | ✅ 做基础版 |
| **叙述者 Narrator** | 把结构化事件转成**文学化叙述**（文风由世界 setting 定基调，可多视角） | ✅ 做 |
| **剧情导演 Plot Director** | 评估故事张力（太平淡？）→ 注入微小环境事件/命运转折维持趣味 | ⚪ 默认关，第二期 |

世界意识维护**世界记忆**——长程的因果链与悬念，用来防止"前面埋的线索后面忘掉"这类剧情断裂。

## 8. LLM 网关与分层路由

所有 LLM 调用的统一入口。

### 职责
- **Provider 抽象**：统一接口，屏蔽 智谱GLM / 通义 / DeepSeek / Moonshot / OpenAI / Anthropic / 本地 Ollama 的差异。
- **分层路由**：按调用层级路由到不同模型。
- **Fallback**：主模型失败 → 备用模型 → 降级。
- **速率限制**：按 provider 分别令牌桶限流，超出排队。
- **结构化输出**：强制 JSON / function calling（ActionProposal、事件、世界构建的结构化产物等都要结构化）。
- **可观测**：token 用量、成本、延迟、失败率统计。
- **缓存**：相同 prompt 复用（prompt caching）。

### 模型分层路由表（每个 World 可覆盖默认配置）

| 调用类型 | 层级 | 推荐模型（示例） |
|----------|------|------------------|
| 世界构建助手 / 世界意识：叙述/裁判/一致性 | **Tier 1 强** | GLM 旗舰 / Claude / GPT-4o |
| 主角色 Agent：决策/对话 | **Tier 2 中** | GLM 主力 / DeepSeek / Qwen |
| 记忆摘要/路人反应 | **Tier 3 弱** | GLM Flash / 本地 Ollama |
| Embedding 向量（检索用） | **专用** | BGE / 智谱 embedding / 本地 |

### 技术选型
- 用 **LiteLLM** 作为网关实现——成熟、覆盖国内模型和 Ollama、自带 fallback/限流/结构化输出/成本统计。
- Embedding 用专门模型，不混用生成模型。
- LLM 密钥用环境变量管理，不入库。

### MVP 策略
架构上完整实现网关（抽象 + 分层路由 + fallback），但**第一版所有 Agent 先用同一个主力模型跑通核心循环**（包括世界构建助手），验证可行后再按上表分层优化。

## 9. 导演介入机制（运行时）

### 四种介入类型（按「软 → 硬」排列）

| 类型 | 例子 | 对涌现的影响 |
|------|------|-------------|
| **注入事件** Inject Event | "一场暴风雨袭来""一个陌生人到镇上" | 软——世界意识和角色自然反应，可能不按预期 |
| **改角色目标** Set Goal | "让艾伦决心复仇" | 软——角色纳入规划，但怯懦者仍可能退缩 |
| **改世界规则/状态** Modify World | "宣布战争爆发""魔法失效" | 硬——确定生效，影响物理引擎和所有人 |
| **强制行动** Force Action | 直接让某角色做某事（跳过阶段①） | 最硬——绕过角色自主性，慎用作"救场" |

### 介入流程（配合阶段③）
```
自动播放中 → 导演点[暂停] → 引擎跑完当前 tick 停下（状态一致）
→ 前端打开介入面板 → 选类型 + 填写 → 创建 DirectorDirective 入队
→ [继续] → 下一个 tick 的阶段③注入：
    注入事件 → 作为 Event 进入世界意识/角色感知
    改目标   → 修改 character.goals
    改规则   → 修改 world.rules / state_flags
    强制行动 → 直接作为该角色的 ActionProposal（跳过①）
```

### 设计原则
**默认鼓励软介入**（注入事件、改目标）——它尊重角色自主性、保留涌现：导演给世界一个"推力"，故事怎么走仍由角色决定。硬介入（改规则、强制行动）作为"救场/转折"工具。所有介入产生的事件在事件流里标记为 🎬 **导演** 来源，与自然事件区分。

> 注意区分：**世界构建助手**是模拟前的"协作定义世界"，**导演介入**是模拟中的"运行时干预"。两者都是导演的权力，但作用阶段和机制不同。

## 10. 技术架构

### 后端（Python / FastAPI）

模块划分（每个模块单一职责，可独立测试）：
```
api/         REST 端点（World 增删查、播放/暂停/调速、介入提交、查询事件流）
ws/          WebSocket（实时推送事件流、叙述、状态变化）
worldbuilder/  世界构建助手（模板、7阶段对话状态机、矛盾检测、健康检查）
engine/      模拟引擎（五阶段循环、调度、暂停控制）
agents/      角色 Agent + 世界意识 Agent（子职责）
physics/     物理引擎（规则、裁决器、数值系统）
llm/         LLM 网关（LiteLLM 封装、分层路由、fallback、限流、成本统计）
memory/      记忆流（存储、检索、embedding）
models/      领域实体（第 4 节定义的实体）
persistence/ 持久化层（SQLite，隔离以便未来迁移）
```

**并发模型**：每个 World 一个 asyncio task 跑循环；阶段①角色并行用 `asyncio.gather` + 网关并发上限；暂停用 `asyncio.Event`（循环在 tick 之间等待，保证状态一致）。世界构建助手是对话驱动的状态机，不在模拟循环内。

### 前端（React + TypeScript + Vite）

主要视图：
- **世界构建向导**：选模板 → 7 阶段对话（每阶段自由聊 + 阶段小结确认 + 可回退）→ 健康度报告 → 开拍
- **主舞台**：实时事件流 + 文学叙述（核心观察区）
- **导演介入面板**：暂停后弹出，四种介入类型
- **角色/关系视图**：角色卡片 + 关系图谱（react-flow）
- **世界状态面板**：时钟、全局状态、规则

状态管理用 Zustand（轻量）；WebSocket 客户端订阅事件流，事件追加到时间线。

### 持久化（MVP 零外部依赖优先）

| 数据 | MVP 选型 | 理由 / 迁移路径 |
|------|---------|--------------|
| 结构化数据（World/Character/Event/Relationship…） | **SQLite** | 零部署、单文件；Event 表带 tick 索引，预留"回放"能力。规模大后迁 PostgreSQL |
| 记忆向量检索 | **内存级检索**（numpy 余弦相似度） | 角色数少、记忆量小够用。规模大后迁 pgvector / Qdrant |

`persistence/` 层隔离，未来迁移 PostgreSQL + pgvector 是平滑的。

### 实时通信
- **WebSocket**：引擎每个 tick 写回后，推送新事件 + 状态 diff 给前端。世界构建对话也走 WebSocket（流式对话）。
- **REST**：控制操作（播放/暂停/调速/介入）走 REST（请求-响应明确）。

## 11. MVP 范围

**MVP 目标**：验证核心循环——*创建一个世界，放几个角色进去，观察他们自主互动产生可读故事，并能暂停介入。*

### ✅ MVP 包含
- **世界构建助手**：模板起点（先做 2-3 个预设模板）+ 7 阶段对话（愿景/世界观/规则/地点/角色/开场/定稿）+ 混合交互（自由对话 + 后台清单 + 阶段小结 + 可回退 + 矛盾检测）+ 定稿健康度检查 → 产出可运行 World
- 模拟引擎五阶段循环（角色层 → 物理引擎 → 导演注入 → 世界意识 → 写回）
- 角色 Agent：感知 → 检索 → 规划 → 提案（含记忆流）
- 世界意识：裁判 + 叙述者 + 基础一致性（剧情导演关）
- 物理引擎：基础可行性校验 + 移动 + 关系数值（战斗/资源数值简化）
- LLM 网关：LiteLLM + 分层路由（先支持 1-2 个 provider，如智谱GLM + 本地）
- 导演介入：四种类型
- 前端：世界构建向导、事件流 + 叙述、介入面板、角色/关系视图、世界状态
- 持久化：SQLite + 内存向量
- 自动播放 + 暂停 + 调速

### ❌ MVP 不做（第二期+）
- 角色 Agent 反思（Reflect）
- 世界意识剧情导演（自动戏剧性）
- 战斗/经济深度数值系统
- 多视角叙述、文风切换
- 回滚/重放（数据层已预留 tick 索引）
- 多用户/账号系统（先单用户）
- 角色视角锁定观察
- 地图可视化等富 UI
- 世界构建的"关键词快速生成"模式（MVP 只做模板+对话）

## 12. 里程碑分解

writing-plans 阶段会从 M1 开始细化成详细任务。每个里程碑都能独立验证。

```
M1 地基        领域模型 + SQLite 持久化 + 基础 API + LLM 网关
               （调通模型、存取数据；无 LLM 业务逻辑也能跑）
M2 世界构建    世界构建助手：模板 + 7阶段对话 + 矛盾检测 + 健康检查
               → 产出可运行的结构化 World 对象
M3 单角色线    用构建好的 World，一个角色 + 最小世界意识(叙述者)
               跑通"感知→决策→叙述"
M4 多角色      多角色并行 + 物理引擎裁决 + 关系网络
M5 介入+前端   WebSocket 实时流 + 导演介入面板 + 主舞台前端
M6 记忆涌现    完整记忆流检索 + 调试涌现质量
```

世界构建（M2）必须早于单角色模拟（M3），因为模拟需要先有一个定义好的 World。
